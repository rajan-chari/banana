What got tested (14 cases)

Used a real http.Server on a kernel-assigned port (listen(0, ...)) plus a real ws client. The fake session is a 35-line EventEmitter stand-in that captures marks / writes / resizes / cleared for assertions.

  • Connect handshake — initial sessions message arrives with current session list.
  • Message routing — input triggers both markUserInput and write; resize forwards cols/rows; clear-input-dirty is wired.
  • Malformed input tolerance — bad JSON and unknown session names don't throw or close the socket.
  • 16ms batch window — two consecutive emit("data", ...) calls coalesce into one data broadcast.
  • Broadcast fan-out — broadcastStatus, broadcastSessionList, broadcastNotification, broadcastName all reach all connected clients.
  • Late session attach — attachSession wires existing clients to a session added after connect.
  • Disconnect cleanup — getClientCount drops, post-disconnect emit("data") doesn't throw.
  • Client count tracks connect/disconnect accurately.
  • Shutdown terminates clients and zeros state.

One hard-won note: the queued message harness

First attempt timed out 13/13 because the ws client emits open then message in rapid succession — by the time the test awaited connect() and then attached a once("message", ...) listener, the initial sessions payload had already fired and been dropped. Real bug in my test setup, would have been a flaky test forever.

The fix is a small queue-from-open-time harness: register the message listener immediately on the WS object, buffer everything into a queue, and have next() drain from the queue with promise-waiters for the empty case. ~25 lines, replaces the racy nextMessage. It's at the top of test/ws-runtime.test.ts if you want to reuse the pattern for any future protocol tests.
